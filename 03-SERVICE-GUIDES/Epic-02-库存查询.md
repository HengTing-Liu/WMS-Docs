## 一、Epic 概述

### 1.1 业务背景

库存查询是仓储业务的核心视图，提供物料、库位、流水等关键信息的实时查询能力，支持库存预警和追溯。

### 1.2 包含模块

| 模块编号 | 模块名称 | Story数 | 开发顺序 |

| WMS0050 | 查询物料 | 4 | 1 |
| WMS0060 | 查询库位 | 4 | 2 |
| WMS0070 | 查询流水 | 4 | 3 |
| WMS0080 | 二维码 | 3 | 4 |

## 二、模块 Story 拆分

### 2.1 WMS0050 查询物料

| Story编号 | Story名称 | 验收标准 | 依赖关系 |
|---------|---------|---------|---------|
| S-0050-01 | 物料库存查询 | 按物料、批次、仓库筛选、显示库存数量 | - |
| S-0050-02 | 物料库存明细 | 查看物料在各个库位的库存分布 | S-0050-01 |
| S-0050-03 | 物料库存导出 | 导出Excel、包含批次信息 | S-0050-01 |
| S-0050-04 | 库存预警设置 | 设置最低库存阈值、预警规则 | S-0050-01 |

**关键字段**：
- 物料编码、物料名称、规格、单位、品牌、批次号
- 仓库编码、库位编码、库位全路径名称
- 现有库存数量、可用库存数量、冻结库存数量
- 生产日期、失效日期

**数据库表**：`inv_inventory`、`material`、`batch`

**权限标识**：`wms:inventory:query`、`wms:inventory:export`

**数据权限**：仓库级数据权限（只能看有权限仓库的库存）

## 三、技术实现要点

### 3.1 库存查询性能优化

```java
// 性能要求：
// - 单表查询 < 200ms
// - 复杂查询（多条件、导出）< 500ms
// - 大数据量（>10000条）分页查询 < 1s

// 优化策略：
// 1. 建立复合索引
// 2. 分页优化（使用游标分页或ID分页）
// 3. 查询结果缓存（热点数据）
// 4. 异步导出（大数据量导出）
```

### 3.2 索引设计

```sql
-- 库存查询索引
CREATE INDEX idx_inv_warehouse_material ON inv_inventory(warehouse_code, material_code);
CREATE INDEX idx_inv_location ON inv_inventory(location_id);
CREATE INDEX idx_inv_batch ON inv_inventory(batch_id);
CREATE INDEX idx_inv_available ON inv_inventory(available_quantity);

-- 流水查询索引
CREATE INDEX idx_change_time_type ON inv_inventory_change(change_time, change_type);
CREATE INDEX idx_change_material ON inv_inventory_change(material_code);
CREATE INDEX idx_change_warehouse ON inv_inventory_change(warehouse_code);
```

### 3.3 缓存策略

| 数据类型 | 缓存策略 | 过期时间 |
|---------|---------|---------|
| 库存快照 | Redis缓存 + 定时刷新 | 5分钟 |
| 库位状态 | 实时查询（变更时主动刷新） | - |
| 流水统计 | 本地缓存 | 1分钟 |
